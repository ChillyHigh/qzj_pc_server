# 执行层实现设计

本文档只描述 `executor/` 如何执行 Plan DAG。Executor 不做几何枚举，不做 IK，不做 TOPPRA，不 import `planner/motions`、`arm`、`chassis`、`trajectory`。

Planner 和 Executor 的唯一交汇点是 `plan/` 包里的 DAG：

```text
planner -> Plan DAG -> executor
```

## 1. 输入契约

Executor 接收一个已经由 Planner 完成的 DAG。DAG 中只处理两类节点：

```python
ActionNode(
    name: str,
    deps: list[AbstractNode],
    kind: "chassis" | "arm" | "flags",
    path: AbstractGeometricPath,
)

WaitNode(
    name: str,
    deps: list[AbstractNode],
    target: FeedbackTarget,
    timeout: float,
)
```

`deps` 是节点引用，不是字符串。`name` 只用于日志、调试和报错。

Executor 每帧从 `ActionNode.path` 读取：

```python
q = path.q(elapsed)
dq = path.dq(elapsed)
duration = path.duration
```

Executor 不知道 path 内部是 TOPPRA、插值、常量保持，还是别的实现。

## 2. 模块职责

```text
executor/
  executor.py       DAG 调度主循环
  mixer.py          active actions -> RobotCommandState

plan/
  types.py          ActionNode / WaitNode / AbstractGeometricPath
  dag.py            children / dep_left / DAG 校验

connection/
  client.py         发送单帧，接收反馈
  protocol.py       14 float + flags + CRC-8
```

依赖方向：

```text
executor -> plan + connection
connection/client.py -> connection/protocol.py
executor 不允许 import planner / arm / chassis / trajectory / toppra
```

## 3. 调度状态

Executor 不使用单个 pointer。DAG 是图，运行时维护几个集合：

```python
nodes: list[AbstractNode]
children: dict[AbstractNode, list[AbstractNode]]
dep_left: dict[AbstractNode, int]

ready: deque[AbstractNode]
active_actions: dict[ActionNode, RunningAction]
active_waits: dict[WaitNode, RunningWait]
done: set[AbstractNode]

kind_busy: dict[str, ActionNode | None]
```

`finish_node(node)` 是唯一推进 DAG 的动作：

```python
done.add(node)

for child in children[node]:
    dep_left[child] -= 1
    if dep_left[child] == 0:
        ready.append(child)
```

## 4. 启动规则

`WaitNode`：

- deps 满足后立即进入 `active_waits`。
- 不占用 `kind`。
- 不输出 frame。
- 只根据 feedback 判断完成。

`ActionNode`：

- deps 满足且 `kind` 空闲后进入 `active_actions`。
- 启动时记录 `start_time`。
- 每帧按 `elapsed = now - start_time` 采样 `path`。
- `elapsed >= path.duration` 后完成并释放 `kind`。

如果 ActionNode deps 满足但 `kind` 被占用，本帧不启动，等待后续 tick 再试。

## 5. WaitNode 语义

WaitNode 是反馈门控，不是串在 Action 后的空动作。

标准写法：

```python
start = AbstractNode(name="start", deps=[])
a = ActionNode(name="chassis_move", deps=[start], kind="chassis", path=chassis_path)
w = WaitNode(name="wait_mid_reached", deps=[start], target=target, timeout=2.0)
b = ActionNode(name="arm_prepose", deps=[w], kind="arm", path=arm_path)
```

执行含义：

```text
start 完成
-> A 启动，底盘开始移动
-> W 启动，开始监听反馈

W 满足
-> B 进入 ready
-> 如果 arm 空闲，B 启动
```

A 和 B 是否同时运行，只看 `kind` 是否冲突。

## 6. 100Hz 主循环

控制频率建议先用 100Hz，可配置。

```python
while not dag_done:
    now = clock()
    feedback = client.getFeedback()

    # 实际写代码并不会建立下面的简单函数，而是直接把逻辑写在这里
    update_active_waits(feedback, now)
    update_active_actions(now)
    start_ready_nodes(now)

    state = mixer.mix(active_actions, hold_state, now)
    client.send_frame(state)

    sleep_until_next_tick()
```

顺序要求：

- 先处理反馈，避免 WaitNode 满足后多等一帧。
- 再释放已完成 Action 的 `kind`。
- 再启动 ready 节点。
- 最后 mix 当前 active actions 并下发一帧。

## 7. RuntimeMixer

RuntimeMixer 每帧从 active actions 合成 `RobotCommandState`。

基础状态来自上一帧目标或当前保持状态：

```text
x, y, yaw,
h, q1, q2, gripper_yaw, gripper_opening,
dx, dy, dyaw,
dh, dq1, dq2,
flags
```

合成规则：

```text
active chassis:
  q -> x/y/yaw
  dq -> dx/dy/dyaw

active arm:
  q -> h/q1/q2/gripper_yaw/gripper_opening
  dq -> dh/dq1/dq2

active flags:
  q -> flags
```

没有 active action 的维度保持上一帧目标。

## 8. Action 完成与 Wait 完成

ActionNode 完成：

```text
elapsed >= path.duration
```

Action 完成只表示轨迹播放结束，不表示下位机到位。

WaitNode 完成：

```text
target.satisfied(feedback) == True
```

到位、吸附成功、漏斗状态确认都必须通过 WaitNode 表达。

## 9. 下发协议

Executor 通过 `connection/client.py` 每个 tick 发送一帧。

帧格式：

```text
SOF(0xAA55) + 14 * float32 + uint8 flags + uint8 crc
```

字段：

```text
x, y, yaw,
h, q1, q2, gripper_yaw, gripper_opening,
dx, dy, dyaw,
dh, dq1, dq2,
flags
```

`x/y/yaw` 使用四主动轮对角线交点。Executor、Client 和反馈判断都不使用底盘几何中心坐标。

`crc` 使用 CRC-8/ATM，`poly=0x07, init=0x00, xorout=0x00`，只覆盖 SOF 后面的 payload，不包含 SOF 和 crc 本身。反馈帧同样在末尾追加 1 字节 CRC；反馈 CRC 错误时丢弃该帧并继续接收，不发布该帧数据。

Client 只负责发送和解析反馈，不理解 DAG、TOPPRA 或 motion。

## 10. 失败处理

失败来源：

- WaitNode timeout。
- feedback failed。
- connection lost。
- protocol error。

处理：

```text
1. 停止启动新的 node
2. 保持或发送安全状态
3. 标记 DAG 执行失败
4. 把 current_state / remaining_tasks 交回上层请求 replan
```

Executor 不自己规划 recovery。Recovery Plan 也必须由 Planner 生成 DAG。

## 11. 测试点

- deps 未满足时节点不会启动。
- WaitNode 和 ActionNode 可从同一个 deps 同时启动。
- WaitNode 满足后能解锁后继节点。
- 同一 `kind` 不会并行执行两个 ActionNode。
- active action 每帧调用 `path.q(t)` / `path.dq(t)`。
- 不预生成整段 frame list。
- WaitNode timeout 能停止 DAG。
- connection 层不 import planner/trajectory。
