# 规划层实现设计

本文档只描述 `planner/` 如何生成 Plan DAG。Planner 不下发帧，不读串口，不等待反馈，不 import `executor` 或 `connection`。

Planner 和 Executor 的唯一交汇点是 `plan/` 包里的 DAG：

```text
planner -> Plan DAG -> executor
```

## 1. 输出契约

Planner 的最终输出是一个 `DAG` 实例。DAG 中只允许两类节点：

```python
AbstractNode(
    name: str,
    deps: list["AbstractNode"],
)

ActionNode(
    name: str,
    deps: list[AbstractNode],
    kind: "chassis" | "arm" | "flags",
    path: AbstractGeometricPath,
)

WaitNode(
    name: str,
    deps: list[AbstractNode],
    target: FeedbackTarget,
    timeout: float,
)
```

`deps` 直接保存节点引用，不用字符串 id。`name` 只用于日志、调试和报错，不作为依赖索引。

`ActionNode` 的业务载荷有且只有 `path: AbstractGeometricPath`。Planner 不能把 `q/dq/duration` 拆成字段塞进 ActionNode。

```python
AbstractGeometricPath(
    duration: float,
    q(t: float) -> np.ndarray,
    dq(t: float) -> np.ndarray,
)
```

## 2. 模块职责

```text
planner/
  motions.py        小 motion 库，生成单个 ActionNode.path
  planner.py        枚举候选、评分、输出 DAG

plan/
  types.py          DAG 节点和 AbstractGeometricPath 契约
  dag.py            DAG / children / dep_left

arm/
  types.py
  five_bar.py
  toppra_planner.py

chassis/
  types.py
  toppra_planner.py

trajectory/
  types.py
  densify.py
  smoothing.py
  toppra_planner.py
```

依赖方向：

```text
planner -> plan + arm + chassis + trajectory
planner/motions.py -> arm + chassis
planner 不允许 import executor / connection
```

## 3. Planner 输入

```python
PlanningInput(
    pickup_assignment: list[int],
    drop_assignment: list[int],
    current_state: RobotState,
    remaining_tasks: RemainingTasks,
)
```

Planner 使用输入决定：

- 取货顺序。
- 放货顺序。
- 三种豆子的容器分配。
- 底盘停车姿态。
- arm 末端姿态和夹爪 yaw / opening。
- 哪些节点并行，哪些节点由 WaitNode 门控。

## 4. Motion 的边界

`planner/motions.py` 只提供小函数。每个函数只生成一个 `AbstractGeometricPath`，不生成多个 ActionNode，不决定 DAG 结构。

例子：

```python
path = motions.chassis_s_cross_half(state, target_pose)
node = ActionNode(
    name="move_to_pick_area",
    deps=[start],
    kind="chassis",
    path=path,
)
```

常见 motion：

```text
chassis_direct_to(x, y, yaw)
chassis_s_cross_half(x, y, yaw)
arm_direct_to(h, endpoint_xy, gripper_yaw, gripper_opening)
arm_cross_half_plane(...)
arm_grip_lift(...)
flags_hold_or_switch(flags)
```

## 5. TOPPRA 位置

TOPPRA 在 Planner 输出 DAG 前完成。

流程：

```text
业务目标
-> motion 生成 arm/chassis Waypoint list
-> arm/chassis toppra_planner
-> AbstractGeometricPath
-> ActionNode.path
-> DAG
```

Executor 不知道 Waypoint、TOPPRA、IK，也不 import `arm/chassis/trajectory`。

## 6. DAG 生成规则

DAG 不是顺序列表。Planner 必须显式写依赖。

反馈门控写法：

```python
start = AbstractNode(name="start", deps=[])

a = ActionNode(
    name="chassis_move_to_mid",
    deps=[start],
    kind="chassis",
    path=chassis_path,
)

w = WaitNode(
    name="wait_mid_reached",
    deps=[start],
    target=chassis_reached(mid_pose),
    timeout=2.0,
)

b = ActionNode(
    name="arm_prepose",
    deps=[w],
    kind="arm",
    path=arm_path,
)
```

含义：

- A 和 W 在 `start` 后同时启动。
- W 只看下位机反馈。
- W 满足后 B 可以接入。
- A 和 B 是否并行由 `kind` 锁决定。

禁止把 WaitNode 串成：

```text
A -> W -> B
```

除非语义确实是“等 A 完成后才开始看反馈”。通常底盘到位等待不是这个语义。

## 7. kind 锁

`ActionNode.kind` 同时就是资源锁：

```text
kind=chassis：x/y/yaw
kind=arm：h/q1/q2/gripper_yaw/gripper_opening
kind=flags：夹爪使能、上漏斗、下漏斗等 flags 状态
```

Planner 生成 DAG 时要避免明显非法并行：

- 精确取放前必须有底盘到位 WaitNode。
- 底盘运动时不能让夹爪伸入箱体。
- 同一 `kind` 的动作可以放进 DAG，但 executor 会按 kind 互斥执行。

## 8. 候选与评分

Planner 可以内部估算 DAG 时间，但估算只用于选最优方案，不是 Executor 的运行依据。

候选生成：

```text
1. 根据抽签得到 bean -> pickup/drop
2. 枚举容器分配
3. 查询取货/放货候选停车位
4. 查询 arm 可达姿态
5. 生成 motion path
6. 构造 ActionNode / WaitNode
7. 生成 DAG
8. 估算 DAG 完成时间并评分
9. 输出最优 DAG
```

评分第一版：

```text
score = estimated_dag_time
```

可以再加：

```text
collision_risk_penalty
yaw_switch_penalty
arm_cross_plane_penalty
```

## 9. 输出前校验

Planner 输出 DAG 前必须校验：

- node id 唯一。
- deps 全部存在。
- 不存在环。
- 每个 ActionNode 都有 `path`。
- `path.q(t)` 维度与 `kind` 匹配。
- 每个需要反馈确认的关键状态都有 WaitNode。
- DAG 中存在 start 和 finish 语义。

Planner 到这里结束。后续调度、反馈、下发帧都属于 Executor。
